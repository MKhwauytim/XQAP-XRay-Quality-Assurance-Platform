### C.2 New algorithm (exact ordering)

```
1. Re-load fresh state (never trust the rendered list):
   fresh = (await loadReferralLog(...)).requests.find(r => r.requestId === request.requestId)
   if (!fresh || fresh.status !== "pending") -> msg_request_already_reviewed; reload; return
2. Verify current ownership:
   current = await loadOrDeriveDistributionCurrent(...)
   notOwned = fresh.xrayImageIds.filter(id => { const e = current?.entries.find(...); 
     return !e || e.assignedTo !== fresh.fromEmployee || (e.status !== "pending" && e.status !== "replacement-requested"); })
   if (notOwned.length > 0) -> abort ALL (atomic semantics) with msg_referral_stale_ownership + ids; no auto-deny. return
3. Idempotent-replay guard:
   alreadyApplied = (await loadDistributionLog(...)).events.some(e => e.sourceRequestId === fresh.requestId)
4. if (!alreadyApplied):
   events = fresh.xrayImageIds.map(id => buildReassignEvent({ ..., sourceRequestId: fresh.requestId }))
   distResult = await appendDistributionEvents(...)   // all-in-one-append = atomic within the log
   if (!distResult.ok) -> error, return                // nothing applied — clean retry
5. updateReferralStatus(..., { status: "approved", ... })
   if (!ok) -> msg_referral_decision_retry (retry skips re-emission via step 3). return
6. Refresh distribution cache, appendWorkspaceAction("referral-approved"), loadData().
```

Same pattern for `handleApproveReplacement`: pending-guard via fresh `loadReplacementLog`, ownership check of `originalXrayImageId`, idempotency via `sourceRequestId` stamped by `executeReplacement` (extend `replacement.ts:130` with optional `sourceRequestId` stamped onto both events; `appendSampleRow` guard already idempotent per `sampleStorage.ts:157`). `handleDenyReferral`/`handleDenyReplacement`: fresh-pending guard only.

Fold ignores unknown fields — `sourceRequestId` flows through; no fold change for Item C.

## Item H — `refreshDistribution` zeroed-snapshot guard (`Population/index.tsx:910-918`)

### H.1 Bug

`refreshDistribution` uses `sampleDrawResult?.rows ?? []` (state). If Phase 4 is reached with unset state while a real sample + log exist on disk, `deriveCurrentDistribution(log, [])` drops every event (`distributionLog.ts:139`) and `saveDistributionCurrent` **persists a zeroed snapshot** + zeroed employee mirrors — visible data loss in employee views until the next correct derive.

### H.2 Fix

```ts
async function refreshDistribution(monthFolderName: string): Promise<void> {
  if (!directoryHandle) return;
  let sampleRows = sampleDrawResult?.rows ?? [];
  const log = await loadDistributionLog(directoryHandle, monthFolderName);

  // Guard: never derive against an empty row set while events exist — fall back to disk.
  if (sampleRows.length === 0 && log.events.length > 0) {
    const master = await loadSampleMaster(directoryHandle, monthFolderName);
    sampleRows = master?.rows ?? [];
    if (sampleRows.length === 0) {
      logError("population:refresh-distribution",
        new Error(`Refusing to persist zeroed distribution.current for ${monthFolderName}`));
      setDistributionMessage({ type: "error", text: getLabels().msg_distribution_refresh_no_sample });
      return; // keep the existing on-disk snapshot untouched
    }
  }

  const current = deriveCurrentDistribution(log, sampleRows);
  setDistributionCurrent(current);
  await saveDistributionCurrent(directoryHandle, monthFolderName, current);
  setMonthRefreshKey((k) => k + 1);
}
```

Also set `logRevision` on the persisted snapshot (mirror `withRevision` pattern, `distributionStorage.ts:181`) — today it persists without, forcing the slow path next load. Verify `loadSampleMaster` import; add `logError`.

Label: `msg_distribution_refresh_no_sample` = `تعذر تحديث حالة التوزيع — لم يتم العثور على عينة محفوظة لهذا الشهر.`

## Item I — safeWrite first-write recovery (`safeWrite.ts:298-305` streamed, `:342-348` small-file)

### I.1 Bug

Both commit paths: on live-verify failure they restore from `.bak` only if a valid `.bak` exists, then throw. On a **first write** (`parsedCurrent === null` → no `.bak`, `:274`/`:320`) a transient live-write failure leaves live corrupt/absent, **verified-good `.tmp` deleted**, data lost. Same hole when `.bak` is unreadable.

### I.2 Fix — promote the verified `.tmp` before giving up (both paths)

Small-file path (`:342-348`):

```ts
if (!verifyOk) {
  const bak = await readText(dir, `${fileName}.bak`);
  if (parseValidJson(bak) !== null) {
    await writeText(dir, fileName, bak as string);
    await removeQuietly(dir, tmpName);
    throw new Error(`Safe-write validation failed for ${fileName}; rolled back to previous version.`);
  }
  // No usable .bak (first write, or .bak corrupt): the staged .tmp WAS verified — promote it.
  const staged2 = await readText(dir, tmpName);
  const staged2Ok = skipVerify ? staged2 === serialized : parseValidJson(staged2) !== null;
  if (staged2Ok) {
    await writeText(dir, fileName, staged2 as string);
    const check = await readText(dir, fileName);
    const checkOk = skipVerify ? check === serialized : parseValidJson(check) !== null;
    if (checkOk) {
      await removeQuietly(dir, tmpName);
      return; // recovered — the write SUCCEEDED via promotion; do not throw
    }
  }
  // Promotion failed too: leave .tmp on disk as the survivor for manual/next-read recovery.
  throw new Error(`Safe-write validation failed for ${fileName}; staged copy kept as ${tmpName}.`);
}
```

Streamed path (`:298-305`): identical structure; re-verify tmp with `verifyStreamedFile(dir, tmpName, stagedInfo)` (info from `:280` in scope), promote by reading `.tmp` text → `writeText` live (if `readText` throws RangeError on an enormous file, catch and fall through to keep-tmp throw), then `verifyStreamedFile(dir, fileName, stagedInfo)`.

State in EDIT_LOG: (1) promotion converts a previously-thrown failure into success; (2) on total failure `.tmp` is now **kept**, not deleted (head comment at `:65` already anticipates leftovers).

Optional hardening (recommended): `safeReadJson` gains a third fallback — if live and `.bak` both invalid but `{file}.tmp` parses as a valid envelope, recover from it (reuse the existing `recoveredFromBak` flag; check `SafeReadResult` consumers before adding a new field).

## Epic 1 — Inter-rater reliability (blind double-review + Cohen's kappa) — FLAGSHIP

### 1.1 Goal
Re-assign a configurable 15–20% of each month's submitted reviews to a second reviewer, blind, and compute Cohen's kappa per reviewer and program-wide, with artifact/ambiguous-image exclusion. Basis: standard IRR practice + ISO/IEC 17020 consistency-of-inspection.

### 1.2 Architecture decision: parallel reliability log — NOT tagged events in the main distribution log
**A fully parallel store under `2-samples/{month}/4-reliability/`. Forced by the code:**
1. `deriveCurrentDistribution` (`distributionLog.ts:145`) keys `entryMap` by `xrayImageId` — one entry per image, last event wins. A second `assigned` event for a completed image would flip the primary entry back to `pending`, corrupting `totalCompleted`, quotas, mirrors, and every report.
2. Answers are keyed `${xrayImageId}::${assignedTo}` in the UI (`XrayReferrals.tsx:365`) and stored per-employee. Writing second reviews there would contaminate `loadAllEmployeeFiles` consumers and double-count work in per-reviewer KPIs.
3. Parallel store = zero report-exclusion plumbing: primary stats never see reliability data.

The main distribution log is never written by this epic. Reliability items appear in a dedicated sub-tab (§1.9), which also makes blindness structurally enforceable.

### 1.5 Selection algorithm (`roundSelection.ts` — pure, deterministic)
1. Pool = entries `status === "completed"` AND matching primary `ItemAnswer` `status === "submitted"` AND not pre-excluded (§1.6 rules applied at selection too).
2. Count = `max(1, floor(targetRate * pool.length))` (0 if pool empty).
3. Draw with `createRng(hashSeedString(selectionSeed))` + `drawWithoutReplacement` from `rng.ts`.
4. Second-reviewer assignment: eligible = active users with `ew.reliabilityReview`, `username !== firstReviewer`, and (by default) WITHOUT view-all-entries (view-all users can read the first answer via `XrayReferrals.tsx:806` readonly path). Admin override widens eligibility → `blindnessCaveat: true` + warning badge. Round-robin over eligible users sorted alphabetically, RNG tie-break.
5. Empty eligible list → `{ ok: false, reason: "rel_no_eligible_reviewers" }`.

**Idempotency:** one round per month in v1. `createReliabilityRound` returns the existing round unchanged if present.

### 1.6 Blindness guarantee and exclusion rules
**Blindness — structural:**
- `InspectionPanel` renders `ReadOnlyView` only when `entry.status === "completed" || savedAnswer?.status === "submitted"` (line 113). The reliability host passes a synthetic `DistributionEntry` (real `row`, `status: "pending"`, `assignedTo: secondReviewer`) → edit mode, empty form.
- `savedAnswer` loads exclusively from `{secondReviewer}.reliability.answers.json`. The first answer never reaches the reliability view's component tree.
- Normal XrayReferrals view cannot leak: reliability assignments are not in `distribution.current.json`/mirrors; second reviewers are (default) non-view-all.
- Advisory caveat (SEC-01 class): first answer is plain JSON on disk — document in module docblock.

**Exclusions (no template changes):** a pair is excluded when, in either review: has-image field (via `DEFAULT_EXEC_FIELD_MAPPINGS.hasImageLabel`) = "no" → `image-unavailable`; image-quality field = "منخفض" → `low-quality`; no verdict extractable from `expertResultFieldId` (normalized via `asSuspicionResult`, `executiveReportData.ts:131`) → `no-verdict`; primary answer's `submittedAt !== assignment.firstSubmittedAt` (reopened via Tier-1 D and resubmitted) → `stale-first-answer`. v1 rule: compare against current `submittedAt`; differ → stale-excluded, surface the count.

### 1.7 Kappa computation (`kappa.ts` — pure)
```ts
export function computeKappa(pairs: Array<{ first: ReviewVerdict; second: ReviewVerdict }>): KappaResult;
// a = both سليمة; d = both اشتباه; b/c = disagreements
// po = (a + d) / n;  pe = ((a+b)(a+c) + (c+d)(b+d)) / n^2
// kappa = (po - pe) / (1 - pe); pe === 1 → kappa = (po === 1 ? 1 : null)
export function kappaBand(k: number): ReviewerKappa["band"];
// Landis & Koch: <0.20 poor, 0.21–0.40 fair, 0.41–0.60 moderate, 0.61–0.80 substantial, >0.80 almost-perfect
```
Per-reviewer kappa pools pairs where the user is first OR second reviewer; `pairsAsFirst`/`pairsAsSecond` reported separately. n < 30 → insufficient-data treatment, never hidden.

### 2.4 Seeding mechanics
`deriveCurrentDistribution` drops events whose id is not in `sampleRows` (`distributionLog.ts:154`) → **seeded rows must be appended to `sample.master.json.rows`** via `appendSampleRow` (CAS, idempotent per `sampleStorage.ts:157`). Downstream (fold, snapshot, mirrors, quotas) treats them as ordinary rows with zero code changes — the indistinguishability requirement.

Orchestrator `src/data/gold/seeding.ts` → `seedGoldAssignments(dir, month, perReviewerCount, seededBy)`, invoked from Population Phase 4 bulk-assign success (manager/admin, feature `population.seed-gold`):
```
1. ensureMonthWritable
2. registry = load gold.seeds.json (create if absent)     // idempotency record
3. candidates = active library items sorted (timesSeeded asc, goldId)
4. currentIds = Set(sampleMaster.rows ∪ population ids)
5. per target reviewer: pick perReviewerCount candidates with goldId ∉ registry AND
   row.xrayImageId ∉ currentIds (collision → skip; never re-key a gold row).
   Mix truth polarities ~50/50 (all-threat seeding is detectable/biasing).
   Deterministic: seeded RNG (sampleMaster.rngSeed + "::gold").
6. per (gold, reviewer) strictly in order, resumable:
   a. appendSampleRow(dir, month, gold.row)               // idempotent
   b. appendDistributionEvents([buildAssignEvent({...})]) with
      sourceRequestId: "req-" + djb2hex("gold::" + month + "::" + goldId)
      // replay-guarded like Tier-1 Item C; OPAQUE HASH so a plaintext "gold-…"
      // key in the log doesn't reveal seeds to casual readers
   c. append GoldSeedRecord (casLoop)
   d. increment timesSeeded (best-effort)
7. void appendWorkspaceAction("gold-seeded", …)
```
Crash between (b) and (c): re-run converges (a/b idempotent, then c). The registry is the source of truth for which IDs are seeds.

**Indistinguishability audit:** referrals table renders from `DistributionEntry.row` (fully populated snapshot); `PanelHeader` shows only id + status; stats strip counts golds as real work (intentional). Accepted second-order channels (documented): gold ids absent from Population→browse for that month; replaced/referred golds → `no-result`.

**Nonconforming unit (auditable choice, stated in the report):** reviewer verdict contradicts the original image result — `imageResultAccurate === false` (`executiveReportData.ts:152`). A unit is inspected only when evaluable (submitted review, extractable verdict, image available). Per port: `accepted` if nonconforming ≤ ac; `rejected` if ≥ re; `accepted-revert-to-normal` in a reduced-plan gap; `inconclusive` if inspected < sampleSize.
```ts
export type LotDecisionOutcome = "accepted" | "rejected" | "accepted-revert-to-normal" | "inconclusive";
export type LotDecision = {
  portName: string; lotSize: number; codeLetter: CodeLetter; severity: InspectionSeverity;
  sampleSize: number; inspected: number; nonconforming: number; ac: number; re: number;

};
export type AqlMonthDecisions = {

  aql: AqlValue; inspectionLevel: InspectionLevelId;
  nonconformityDefinition: "reviewer-vs-image-result";

};
```
Stored at `2-samples/{month}/1-main/aql.decisions.json` (month-lock guarded; recomputable while open). **Finalize** ("اعتماد نتائج AQL", manager/admin) advances switching state — idempotent via `AqlProgramState.finalizedMonths`; Archive close dialog gets non-blocking `aql_close_warning_unfinalized`.

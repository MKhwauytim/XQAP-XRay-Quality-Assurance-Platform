## v59.8 — 2026-07-22 — Fix: slide counter RTL reversal, Space-key fullscreen exit, and print-toggle visibility during slideshow

Final whole-branch review fixes for three findings against the merged deck2 slideshow presentation
mode (v59.0–v59.7).

**Finding 1 (Important) — slide counter rendered reversed.** `<span class="deck-slide-counter"
id="deck-slide-counter">` is a direct child of `<body dir="rtl">` and its text is set at runtime to
`"N / M"` (e.g. `"3 / 10"`). Without its own `dir` attribute it inherited RTL from `<body>`, so the
numeral/slash string rendered visually reversed (`"10 / 3"`). This exact bug class was already hit and
fixed elsewhere in this same file family — `.v2-page-foot`, `.v2-stage-port-figure`, `.v2-toc-range`,
and the variant-switcher label all carry `dir="ltr"` for the same reason (see the `.v2-stage-port-
figure` comment in `theme.ts`, documenting EDIT_LOG v40.7's "1 / 4" → "4 / 1" bug). Added `dir="ltr"`
to the counter span's markup and a regression test asserting it.

**Finding 2 (Important) — Space key could exit fullscreen instead of advancing.** Entering fullscreen
is triggered by a mouse click on `#deck-fullscreen-button`. Chromium keeps a clicked button focused
after the click, and `requestFullscreen()` doesn't blur it. The keydown handler treats `' '` (Space)
as "advance to next slide" but a focused `<button>` natively activates on Space too — and the
fullscreen button's own click handler calls `exit.call(document)` when already in fullscreen. So the
first Space press likely both advanced a slide AND exited fullscreen. Fixed both halves: `sync()` now
calls `button.blur()` once fullscreen becomes active (so the button never stays the focused element
during a presentation), and the keydown handler now calls `e.preventDefault()` for every key it
handles (ArrowLeft/PageUp/ArrowRight/Space/PageDown) before calling `goTo(...)`, so Space/PageDown/
PageUp never trigger native browser scroll/button-activation behavior while in fullscreen.

**Finding 3 (Minor/borderline-Important) — per-slide print-toggle stayed visible during the
presentation.** `.slide-controls` (the small print-include toggle switch, top-left of each slide) was
already excluded from the click-to-advance handler (v59.6) but was never hidden by the fullscreen CSS,
so it stayed visible and clickable throughout a presentation — unpolished slideshow chrome. Added
`body.deck-fullscreen .slide-controls{display:none;}` alongside the existing `.srev-footer` hide rule.

## v59.6 — 2026-07-22 — Fix: slideshow scroll-restore firing on load + print-toggle click conflict

Task-review fix for two Important findings against v59.5's `DECK_FULLSCREEN_SCRIPT`.

**Bug 1 — unconditional scroll-restore on page load.** `sync()` is called once, unconditionally,
at the end of the script's IIFE, on every page load (not only after a real fullscreen exit).
Because `current()` is falsy before the user ever enters fullscreen, that initial call fell into
`sync()`'s `else` branch, which unconditionally ran `slides[activeIndex].scrollIntoView({ block:
'start' })`. With `activeIndex` starting at `0`, every fresh page load silently scrolled to slide 0
(the cover) — overriding the browser's native URL-fragment scroll, e.g. a link to
`#slide-recommendations` from the report's own side-nav (`DECK_NAV_SCRIPT`). Fixed with an
`everActivated` flag, set only once a real fullscreen session has been entered (inside `sync()`'s
`if (active)` branch, alongside where `activeIndex` gets recomputed from viewport geometry); the
`else` branch's `scrollIntoView` restoration now only runs when `everActivated` is true, so it fires
on a genuine "just exited fullscreen" transition and never on initial load.

**Bug 2 — click-to-advance conflicts with the per-slide print-include toggle.** Every slide already
renders a print-include checkbox (pre-existing feature, unrelated to this task) as a direct child of
`.slide`, wrapped in a `<div class="slide-controls">` (`slideControls()` in `slides.ts`, styled in
`theme.ts`). The new document-level click-to-advance handler's exclusion list
(`.btn-slide-nav, .deck-slide-counter, #deck-fullscreen-button`) didn't cover it, so clicking that
toggle during a slideshow both flipped print-inclusion state and advanced the slide. Confirmed the
real class from `slides.ts`/`theme.ts` (`.slide-controls` is the positioned wrapper `slideControls()`
renders around `printToggle()`'s `.slide-print-toggle` label, and, in dev-preview builds, the
variant switcher) and added it to the `closest()` exclusion list.

## v59.5 — 2026-07-22 — Add: true slideshow presentation mode for deck2 fullscreen

Task 3 of 3 (final) of the deck2 slideshow-mode plan — the point where the behavior actually goes
live. Task 1 added the `expand`/`compress` icons; Task 2 added the single-slide-presentation CSS
contract (`.deck-slide-active`, `#deck-slide-prev`/`#deck-slide-next`/`#deck-slide-counter`,

`aria-pressed`), and the prev/next buttons get two new configurable Arabic labels

  var button = document.getElementById('deck-fullscreen-button');
  if (!button) return;
  var root = document.documentElement;
  var request = root.requestFullscreen || root.webkitRequestFullscreen;
  var exit = document.exitFullscreen || document.webkitExitFullscreen;
  function current(){ return document.fullscreenElement || document.webkitFullscreenElement; }
  function disable(){
    document.body.classList.remove('deck-fullscreen');
    button.hidden = true;
  }
  if (typeof request !== 'function' || typeof exit !== 'function') { disable(); return; }

  var slides = Array.prototype.slice.call(document.querySelectorAll('.slide'));
  var activeIndex = 0;
  var prevBtn = document.getElementById('deck-slide-prev');
  var nextBtn = document.getElementById('deck-slide-next');
  var counter = document.getElementById('deck-slide-counter');
  var hideTimer = null;

  function showControls(){
    document.body.classList.add('deck-controls-visible');
    if (hideTimer) clearTimeout(hideTimer);
    hideTimer = setTimeout(function(){ document.body.classList.remove('deck-controls-visible'); }, 2500);
  }

  function renderSlide(){
    for (var i = 0; i < slides.length; i++) {
      slides[i].classList.toggle('deck-slide-active', i === activeIndex);
    }
    if (counter) counter.textContent = (activeIndex + 1) + ' / ' + slides.length;
    if (prevBtn) prevBtn.disabled = activeIndex === 0;
    if (nextBtn) nextBtn.disabled = activeIndex === slides.length - 1;
  }

  function goTo(index){
    if (index < 0 || index >= slides.length || index === activeIndex) return;
    activeIndex = index;
    renderSlide();
  }

  function sync(){
    var active = Boolean(current());
    var label = button.getAttribute(active ? 'data-exit-label' : 'data-enter-label');

      var idx = 0;

        if (slides[i].getBoundingClientRect().top <= thresholdY) idx = i; else break;
      }
      activeIndex = idx;

    button.setAttribute('aria-pressed', active ? 'true' : 'false');
    button.setAttribute('aria-label', label);
    button.setAttribute('title', label);

  });
  document.addEventListener('fullscreenchange', sync);
  document.addEventListener('webkitfullscreenchange', sync);
  document.addEventListener('fullscreenerror', disable);
  document.addEventListener('webkitfullscreenerror', disable);

  if (prevBtn) prevBtn.addEventListener('click', function(e){ e.stopPropagation(); goTo(activeIndex - 1); showControls(); });
  if (nextBtn) nextBtn.addEventListener('click', function(e){ e.stopPropagation(); goTo(activeIndex + 1); showControls(); });

## v59.4 — 2026-07-22 — Fix: fullscreen button was structurally hidden by its own toolbar CSS

Task-review fix for v59.3's fullscreen CSS. `body.deck-fullscreen .deck-toolbar>*{display:none;}`
was written assuming `#deck-fullscreen-button.btn-fullscreen` is a direct child of `.deck-toolbar`,
but the real markup (`index.ts`) nests it inside a `.deck-toolbar-actions` wrapper div alongside the
theme-toggle label and the print button — `.deck-toolbar-actions` is the actual direct child. The
`>*` selector therefore matched and hid `.deck-toolbar-actions` itself, taking the fullscreen button
down with it via `display:none` on an ancestor — a state no descendant rule can undo, so the

## v59.0 — 2026-07-22 — Add: replacement-candidate index so opening the replacement dialog no longer reads the full population

**Governance note — deliberate, recorded exception.** The 2026-07-22 data-management audit
classifies replacement-candidate storage changes as "proposal-covered — Phases A/C/D" of
`docs/architecture/LARGE_POPULATION_PERFORMANCE_PROPOSAL_2026-07-22.md`, and the v57.8 entry below
states the replacement candidate design was "explicitly held for the proposal's approved phase
sequence rather than implemented as ad hoc storage changes." The product owner consciously excepted
this one piece from that hold: it is a narrow, self-contained index for exactly one read path, not
the general port-partitioned rewrite of `population.final.json` that Phase C defines for every
consumer. This entry is that record, so the audit's classification is not silently contradicted and
a future Phase C implementer knows this index exists and predates the general mechanism.

**What changed.** Opening the replacement dialog previously read the complete
`population.final.json` (400,000+ rows in real workspaces) to compute a candidate list that ends up
capped at 100 rows — the single largest avoidable read on the employee-facing path. `getReplacementCandidates()`'s
existing `{ recommended, all }` split (same tier+stage+port = immediate use; broader/cascaded
matches = routed through the existing referral-approval pipeline) is unchanged business logic.

A new replacement-candidate index is built once at population-save time from the `processedRows`
already in memory (zero extra population read), split into up to 10 files per month — one per
(CertScan tier × stage key, including the "unknown" stage) — under
`1-population/{month}/2-processed/replacement-index/`, each holding full `PreparedPopulationRow`
objects (a lightweight projection cannot be used for the immediate-execution path, which splices the
chosen row directly into `sample.master.json`). A small manifest records the source population's
envelope revision and a hash of the fully-resolved stage mappings (stage aliases can be edited
independently of population processing, so the revision alone cannot detect that drift); either
mismatch, or a missing index (legacy months processed before this feature existed), triggers a
fallback to the unchanged full-population read plus a non-blocking background rebuild — never a
population re-save cost on the interactive path.

Bucket contents preserve original population row order (never re-sorted): the existing capSeeded
Fisher-Yates draw is order-sensitive, so this had to produce byte-identical output to the old
full-scan path for the same inputs, not just similar results. The cascade fallback (when the primary
tier+stage bucket is empty after dedup) compares POST-DEDUP supply across sibling buckets, never the
manifest's raw row count — a stage can be almost entirely pre-sampled (e.g. "first" under the
default sampling rules) while still showing a large raw count, which would otherwise pick the wrong
cascade winner. Manifest publication is monotonic (mirrors `updateMonthStatus`'s CAS pattern): a
rebuild is skipped entirely, before any bucket file is touched, if the current manifest's revision is
already at or ahead of it — closing a real bug caught during TDD where a stale rebuild was still
clobbering bucket file *contents* even though the manifest correctly stayed pinned to the newer
revision.

Regression coverage includes an equivalence suite proving the indexed path returns identical
candidate ids in identical order to the pre-existing full-scan path across a normal case, a cascade
case, and an oversized-pool (>100) capping case — the one that actually exercises Fisher-Yates
order-sensitivity — plus the post-dedup-vs-raw-count cascade trap, staleness/fallback triggers, and
index-build failure resilience (a failure here must never sink an otherwise-successful population
save). `replacement.test.ts` passes unmodified, confirming the primitives extracted from it
(`buildExclusionSets`, `isEligibleCandidate`, exported `capSeeded`/`REPLACEMENT_POOL_LIMIT`) preserve
its exact behavior.

**Manual verification note.** The interactive click-through (connect a real workspace → process →
sample → distribute → open the replacement dialog) requires a native OS folder-picker gesture that
browser automation cannot drive — the same limitation noted when reviewing PR #37 earlier today. Verified

1. The monotonic rebuild guard compared only `sourceRevision`, so an admin editing stage-mapping

## v57.4 — 2026-07-22 — Fix (backup): make restorable backups JSON-first and bound optional XLSX memory

Owner-reported: workspace backups could take an extremely long time and sometimes fail with
`Invalid array length`. The backup path copied the complete authoritative JSON snapshot and then,
before declaring success, loaded every month again to rebuild redundant XLSX copies in 250,000-row
SheetJS workbooks. Real workspaces exceed 400,000 rows. SheetJS materializes worksheet cells and
the XLSX ZIP in browser memory, so wide datasets could require millions of live cell objects plus
serialization buffers and exhaust the JavaScript array/heap limits.

**Behavior change:** routine manual, automatic, and pre-restore backups now create the complete,
restorable JSON snapshot only. XLSX files are convenience exports, not restore inputs, and are no
longer generated silently as part of every backup. The Archive screen exposes a clearly labeled,
default-off Arabic checkbox for users who explicitly need XLSX copies of smaller datasets.

The compatibility XLSX path now limits a dataset to 100,000 rows, limits each workbook part to
25,000 rows and approximately 250,000 cells, and collects headers incrementally instead of using a
large `flatMap` allocation. Oversized or failed XLSX exports produce a clear Arabic warning while
the completed JSON backup is still recorded successfully. Existing backup manifests retain their
XLSX fields (empty in new JSON-only backups), and history continues to read legacy manifests with
populated XLSX metadata.

Regression coverage verifies the JSON-only default, small explicit XLSX exports, immediate
400,000-row rejection before worksheet allocation, missing-month behavior, and legacy manifest
history compatibility.

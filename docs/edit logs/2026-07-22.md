## v59.24 — 2026-07-22 — Change (population): defer risk/BI raw-import reads until a month is actually unprocessed

The overnight session's optimization item (A1 in the large-population performance
brief) — the piece that most directly targets the user's original "connect to the
latest month, don't eagerly load everything" scaling request. `loadMonthForEditing`
(the loader every month selection runs through) unconditionally read both
`risk.raw.json` and `bi.raw.json` in its opening `Promise.all`, regardless of
whether anything downstream would use them. In real workspaces these are the two
largest files per month (up to the full 200k-400k row population each); nothing in
phase derivation, sampling, distribution, or Browse ever reads `riskRawRows`/
`biRawRows` (confirmed via `buildLoadedMonthState`, which only inspects
`distributionCurrent`/`sampleData`/`populationRows`) — only Phase 1/2's own display
of the originally-uploaded workbook needs them, and only while the month is still
awaiting processing.

**What changed.** The manifest is now read first (small, fast — one JSON file, not
hundreds of thousands of rows), and the raw-file reads are skipped entirely once
`manifest.status` has advanced past `"raw-saved"` (i.e. `"processed-saved"`,
`"sampled"`, `"distributed"`, or `"closed"` — every status where
`population.final.json` already exists and is what everything downstream actually
uses). A missing or unreadable manifest keeps the previous always-attempt behavior
unchanged — this is a demand-reduction optimization, never a new failure mode: on
any uncertainty, it reads exactly what it always read before. Every other field
`loadMonthForEditing` returns (`populationRows`, `certScanRows`,
`processingSummary`, `sampleData`, `distributionCurrent`, `manifest`) is completely
unaffected; `riskRawRows`/`biRawRows` simply come back as `[]` instead of the full
arrays, which is exactly what `MonthEditData`'s own type already allows and what
every existing consumer of those two fields already null/empty-checks.

**Known, deliberate scope boundary (not a regression left half-fixed).** One
existing capability narrows slightly: a user who reprocesses an already-processed
month *without* re-uploading a fresh Excel file (reusing the month-load's
auto-populated workbook) will now see `handleProcessPopulation`'s existing "ارفع
ملف وكالة المخاطر أولاً" guard fire, since the raw workbook is no longer
auto-loaded for that case, and must upload again. Wiring a full on-demand
raw-refetch for that one specific path (the `loadMonthTokenRef`-guarded fetch the
performance brief separately named as future work) was deliberately left as a
follow-up rather than rushed in tonight alongside everything else, per the brief's
own "drop the STRETCH scope the moment it stops being purely mechanical" guidance
for this exact class of item — this repo's session-hydration and month-load race
guards are hard-won and not something to extend casually late at night. Every
*other* consumer (phase derivation, sampling, distribution, Browse, the Reports
tab's sample-report generation, which also called this same loader and never used
the raw fields at all) is unaffected and gets the full benefit immediately.

Verified live in the browser against the demo workspace's already-processed month
(200-row population, 60-item sample, Phase 4 distribution view) before and after:
identical rendering, zero console errors.

## v59.23 — 2026-07-22 — Fix (reports): tsc -b spread-tuple build error in a Wave 2 mock

Post-landing review correction. `npm run build` (`tsc -b`, project-references build mode) failed where `npm run typecheck` (`tsc --noEmit`) had passed clean: `runPowerBiExport: (...args: unknown[]) => pbiExportMock.impl(...args)` -- spreading a plain `unknown[]` rest parameter into `pbiExportMock.impl`, whose type is inferred by `vi.fn()` from a zero-argument implementation, tripped TS2556 ("a spread argument must either have a tuple type or be passed to a rest parameter") under `tsc -b`'s stricter checking. ## v59.22 — 2026-07-22 — Fix (population): wizard bulk-assign permission flag, CertScan quota lock, and render-time phase-button gating

Bucket B13 (population wizard gating): three related permission-gating fixes in the Population wizard (`src/components/Sidebar/Tabs/Population/`) — a bulk-assignment button wired to the wrong permission flag, CertScan quota fields with no permission gating at all, and several phase-transition buttons (file upload, process, export, draw-sample) missing the render-time permission/closed-month/loading gating that Phase 4's distribution actions already had.

1. **Bulk-assignment wrong-flag gate** (`PhaseFourDistribution.tsx`, `index.tsx`, synthesis high, CONFIRMED via live review). The "تطبيق وحفظ التوزيع التلقائي" (bulk-assign) button was disabled/enabled off `canDistribute` (the per-row `distribute-samples` permission). Shipped supervisor defaults grant `bulk-assign: true` but `distribute-samples: false`, so a supervisor who explicitly has bulk-assign permission still saw the button wrongly disabled. Added a dedicated `canBulkAssign` prop (`canMutate('bulk-assign')`, also withdrawn on closed-month/month-loading like its sibling flags) threaded from `index.tsx` into `PhaseFourDistribution`, gating the bulk button's `disabled` + `title` and the `handleRunBulkAssignment` handler independently of `canDistribute`. Per-row assign/reassign/complete/replace actions (`DistributionRow`) keep using `canDistribute`, unchanged.

2. **CertScan quota fields ignored lock+permission** (`PhaseThreeSampling.tsx`, synthesis medium, CONFIRMED via live review). The per-stage "نوع كوتا CertScan" select and "القيمة" input had no `disabled` attribute at all — a locked stage or a role without `configure-sample` could still edit them client-side, even though the change would be silently rejected (and previously invisible) by the existing `handleConfigChange` handler check in `index.tsx`. Added `disabled={isLockedState || !canConfigureSample}` to both fields, matching the sibling "طريقة السحب"/"القيمة المطلوبة" fields' lock-based disabling plus the missing permission check (scoped to the CertScan fields only, per the finding's line reference — the siblings' own gating is unchanged). Threaded `processingMessage` into `PhaseThreeSampling` and rendered it (`role="status"`) so a permission-denied edit is now visible instead of silently reverting on the next render.

3. **Render-time permission gating for phase-transition buttons** (`PhaseOneUpload.tsx`, `PhaseTwoReportAndProcessing.tsx`, `PhaseThreeSampling.tsx`, `index.tsx`, synthesis medium). Phase 1's file-picker cards, Phase 2's process/export buttons, and Phase 3's draw-sample button had no (or an incomplete) render-time permission/closed-month/loading gate, unlike Phase 4's existing `canDistribute` pattern — the underlying handlers already rejected these actions, but the buttons stayed clickable/enabled. Extracted a `computeWizardCapabilities` helper in `index.tsx` (also keeps `PopulationTab` under the repo's function-line/complexity budget, `npm run check:complexity`) that derives `canUploadNow`/`canProcessNow`/`canExportNow` (permission + closed-month + month-loading; export is excepted from closed-month since it doesn't write to the workspace, matching `handleExportPopulation`'s existing handler check) alongside the pre-existing `canDrawSample`/`canDistributeSamples`/`canBulkAssign`. Threaded `canUpload` into `PhaseOneUpload` (file-picker cards wrapped in a gated container — `FileUploadCard` itself exposes no `disabled` prop and is owned by a different bucket), `canProcess`/`canExport` into `PhaseTwoReportAndProcessing`, and `canDrawSample`/`canConfigureSample`/`processingMessage` into `PhaseThreeSampling`'s draw button — each with a matching denial `title`, mirroring Phase 4's convention.

## v59.19 — 2026-07-22 — Fix (population): Browse filter single-select collapse, stale BI/risk report join key, and normalizer perf

Bucket B12 (population data layer): four fixes across the Browse view, the exported processing report's BI/risk comparison, the risk/BI Excel-import normalizers, and a small dead-code dedup.

1. **Browse per-column filter single-select collapse** (synthesis high, confirmed via live testing). The open column's filter dropdown built its option list from `filteredRows` — rows already narrowed by *every* active column filter, including the open column's own filter. So the instant a user checked one value, the option list collapsed to show only that value, hiding every other legitimate option. Fixed by building the option list from rows filtered by every *other* column's filter (and search) while excluding the open column's own filter via the existing `exceptKey` parameter of `rowMatchesColumnFilters`.

2. **Exported processing-report join key**. The BI/risk comparison report (`buildBiRiskComparisonReport`) joined risk and BI rows on a bare, upper-cased `xrayImageId` via a local `makeComparisonKey`. Two rows sharing an X-ray ID across two *different* ports collapsed into a single `Map` entry per side, silently under-counting `totalMatchedRecords`. Replaced with the shared `makeBiMatchKey(xrayImageId, portName)` from `processing/populationProcessor.ts` — the same key `populationProcessor.ts` and `DataAccuracyReport.tsx` already use — and deleted the stale `xrayScanId`/`xrayImageID` field probes (normalized risk/BI rows only ever carry `xrayImageId`), both in `makeComparisonKey` and in the `sampleDifferentRows.push(...)` call.

3. **Normalizer header-lookup hoist**. `normalizeRiskRow`/`normalizeBiRow` called `getFirstAvailableValue(sourceRow, ...)` once per output field (~27 fields for risk, ~29 for BI), and that function rebuilt the entire header-lookup `Map` from scratch on every call — an O(fields × columns) rebuild per row. `createHeaderLookup(sourceRow)` is now called once at the top of each normalizer function, and the resulting `Map` is threaded through every `getFirstAvailableValue` call. Purely a plumbing change — the candidate-header resolution order and fallback behavior are unchanged.

4. **`createEmptyStageCounts` dedup**. `components/helpers.ts` re-exported `createEmptyStageCounts` from `data/population/stageHelpers.ts`, but nothing in the codebase imported it that way — `reportDataBuilder.ts` instead carried its own byte-identical duplicate implementation. Dropped the unused re-export from `helpers.ts` (kept `getStageKey`/`formatStageLabel`, which are still consumed elsewhere) and pointed `reportDataBuilder.ts` at the shared `stageHelpers.ts` implementation directly.

## v59.11 — 2026-07-22 — Fix (auth): startup session-hydration race dropped valid managed-user sessions

**What changed / why:** AuthGate's `getInitialSession()` ran synchronously at mount (via `useState(getInitialSession)`), before `WorkspaceProvider`'s async bootstrap effect had a chance to load the persisted workspace and sync its `users.permissions.json` into the in-memory user-management runtime state (`applyDiskUsers`, called only after `checkWorkspaceStructure` resolves "ready" *and* a further `loadWorkspaceFiles` await). At that point `getManagedLoginUsers()` reflects only the 5 hardcoded default seed users (`runtimeUserManagementState` starts `null` on every fresh module load), so a persisted `sessionStorage` session for any real, disk-backed managed user (anyone other than those 5 seed users) failed the `userStillExists` check and was immediately wiped via `clearSession()` — even though the workspace, moments later, would have confirmed the user is real. Since nothing ever *restores* a session once React state holds `null`, the user was permanently bounced to the login screen on every reload despite having a valid session.

Fix: defer the CLEAR decision, not the READ. `getInitialSession()` now just returns the persisted session as-is. A new `usersHydrated` boolean on `WorkspaceContextValue` (supplied by `WorkspaceProvider`, `false` until `applyDiskUsers` has actually run for the current workspace connection, reset to `false` in `clearWorkspace()`) gates a new `AuthGate` effect that performs the real "does this user still exist" check once hydration is confirmed — reusing the same check (extracted into `stillHasManagedUser`) the pre-existing `subscribeToUserManagementChanges` handler already used for live post-mount changes.

Also completed the requested low-risk dead-code cleanup: `loadWorkspaceFiles()` no longer reads `workspace.manifest.json` / `sample.master.json` / `sample.distribution.json` from disk — nothing ever consumed those three `WorkspaceLoadedFiles` fields, and the latter two paths don't even exist under the current numbered layout (`2-samples/{month}/...`), so those reads failed as "missing" on every single workspace load. `usersPermissions`, which `applyDiskUsers` does consume, is untouched. The `WorkspaceLoadedFiles` *type* itself still declares all 4 fields (always `null` for the 3 dead ones) because that type lives in `src/data/workspace/workspaceTypes.ts`, outside this bucket's owned_files — see the deferred-task note.

Sub-tab role ceilings never enforced.** `TAB_ROLE_CEILINGS` was built only from top-level tabs (`!tab.parentId`), so a sub-tab's own `allowedRoles` in `TAB_CATALOG` (e.g. `reports/kpi`) was pure decoration — nothing ever read it. Added a companion `SUB_TAB_ROLE_CEILINGS` map plus a unified `roleCeilingFor(tabId)` helper (checks both levels via the existing `CATALOG_BY_ID`), and wired it into the three real enforcement points: App.tsx's sidebar sub-tab filter, `usePermissions.canAccessTab` (which `TabGuard` calls into, so it inherits the fix for free), and `PermissionSections.tsx`'s `isCeilingLocked`. That last one had a live bug: the sub-tab row's lock check passed the *parent* tab's id (`tab.id`) instead of the sub-tab's own id (`sub.id`), so the admin matrix UI showed the wrong lock state for every sub-tab whose ceiling differs from its parent's. Because `reports/kpi`'s ceiling was previously `["manager","admin"]` while the shipped permission-matrix defaults already grant supervisors "view" (working, tested behavior), turning on real enforcement without widening the ceiling would have regressed supervisors — so `reports/kpi` widened to `["supervisor","manager","admin"]` to match tested reality; `report-designer` already excluded guest and needed no change.

**2. post-notification cascaded against the wrong tab.** `TAB_FEATURE_MAP['employee-workspace']` included `post-notification`, but the feature is only ever rendered on the `ew/notifications` top-level tab (`NotificationManager`, mounted from `NotificationCenter/index.tsx`). Moved the entry to a new `TAB_FEATURE_MAP['ew/notifications']` key.

**3. ew/inspection-form was a guaranteed dead end for employee/supervisor.** Both roles default to `manage-inspection-template: false`, so `TemplateBuilder` (rendered at that sub-tab) always shows its access-denied empty state for them — yet the sidebar entry and page-level "view"/"edit" access were still granted. Set default `ew/inspection-form` access to `none` for both roles so the guaranteed-blocked sidebar entry disappears; manager (which has the feature on) keeps `edit`.

**4. Decorative feature toggles.** `export-archive` and `view-employee-stats` had zero consumers anywhere in `src/` (grep-confirmed) — pure decoration in the admin Feature Permissions matrix. Removed both from `MANAGED_FEATURE_GROUPS`, `TAB_FEATURE_MAP`, and `FEATURE_DEFAULTS` (their removal from `ALL_FEATURE_IDS`/`createDefaultFeaturePermissions()`/`FEATURE_MUTATION_STORAGE_LOOKUP` is automatic, all three derive from the groups list). **5. A2 employee landing tab.** When no explicit/valid tab selection exists, employees now prefer `employee-workspace` (when present in `allowedTabs`) over whatever tab happens to sort first.

**6. Gate widened to `admin || manager`. Did not touch `backupStorage.ts` (owned by another bucket) or Archive's copy text.

## v59.8 — 2026-07-22 — Fix: slide counter RTL reversal, Space-key fullscreen exit, and print-toggle visibility during slideshow

Final whole-branch review fixes for three findings against the merged deck2 slideshow presentation
mode (v59.0–v59.7).

**Finding 1 (Important) — slide counter rendered reversed.** `<span class="deck-slide-counter"
id="deck-slide-counter">` is a direct child of `<body dir="rtl">` and its text is set at runtime to
`"N / M"` (e.g. `"3 / 10"`). Without its own `dir` attribute it inherited RTL from `<body>`, so the
numeral/slash string rendered visually reversed (`"10 / 3"`). This exact bug class was already hit and
fixed elsewhere in this same file family — `.v2-page-foot`, `.v2-stage-port-figure`, `.v2-toc-range`,
and the variant-switcher label all carry `dir="ltr"` for the same reason (see the `.v2-stage-port-
figure` comment in `theme.ts`, documenting EDIT_LOG v40.7's "1 / 4" → "4 / 1" bug). Added `dir="ltr"`
to the counter span's markup and a regression test asserting it.

**Finding 2 (Important) — Space key could exit fullscreen instead of advancing.** Entering fullscreen
is triggered by a mouse click on `#deck-fullscreen-button`. Chromium keeps a clicked button focused
after the click, and `requestFullscreen()` doesn't blur it. The keydown handler treats `' '` (Space)
as "advance to next slide" but a focused `<button>` natively activates on Space too — and the
fullscreen button's own click handler calls `exit.call(document)` when already in fullscreen. So the
first Space press likely both advanced a slide AND exited fullscreen. Fixed both halves: `sync()` now
calls `button.blur()` once fullscreen becomes active (so the button never stays the focused element
during a presentation), and the keydown handler now calls `e.preventDefault()` for every key it
handles (ArrowLeft/PageUp/ArrowRight/Space/PageDown) before calling `goTo(...)`, so Space/PageDown/
PageUp never trigger native browser scroll/button-activation behavior while in fullscreen.

**Finding 3 (Minor/borderline-Important) — per-slide print-toggle stayed visible during the
presentation.** `.slide-controls` (the small print-include toggle switch, top-left of each slide) was
already excluded from the click-to-advance handler (v59.6) but was never hidden by the fullscreen CSS,
so it stayed visible and clickable throughout a presentation — unpolished slideshow chrome. Added
`body.deck-fullscreen .slide-controls{display:none;}` alongside the existing `.srev-footer` hide rule.

## v59.6 — 2026-07-22 — Fix: slideshow scroll-restore firing on load + print-toggle click conflict

Task-review fix for two Important findings against v59.5's `DECK_FULLSCREEN_SCRIPT`.

**Bug 1 — unconditional scroll-restore on page load.** `sync()` is called once, unconditionally,
at the end of the script's IIFE, on every page load (not only after a real fullscreen exit).
Because `current()` is falsy before the user ever enters fullscreen, that initial call fell into
`sync()`'s `else` branch, which unconditionally ran `slides[activeIndex].scrollIntoView({ block:
'start' })`. With `activeIndex` starting at `0`, every fresh page load silently scrolled to slide 0
(the cover) — overriding the browser's native URL-fragment scroll, e.g. a link to
`#slide-recommendations` from the report's own side-nav (`DECK_NAV_SCRIPT`). Fixed with an
`everActivated` flag, set only once a real fullscreen session has been entered (inside `sync()`'s
`if (active)` branch, alongside where `activeIndex` gets recomputed from viewport geometry); the
`else` branch's `scrollIntoView` restoration now only runs when `everActivated` is true, so it fires
on a genuine "just exited fullscreen" transition and never on initial load.

**Bug 2 — click-to-advance conflicts with the per-slide print-include toggle.** Every slide already
renders a print-include checkbox (pre-existing feature, unrelated to this task) as a direct child of
`.slide`, wrapped in a `<div class="slide-controls">` (`slideControls()` in `slides.ts`, styled in

## v59.0 — 2026-07-22 — Add: replacement-candidate index so opening the replacement dialog no longer reads the full population

classifies replacement-candidate storage changes as "proposal-covered — Phases A/C/D" of
`docs/architecture/LARGE_POPULATION_PERFORMANCE_PROPOSAL_2026-07-22.md`, and the v57.8 entry below
states the replacement candidate design was "explicitly held for the proposal's approved phase
sequence rather than implemented as ad hoc storage changes." The product owner consciously excepted
this one piece from that hold: it is a narrow, self-contained index for exactly one read path, not
the general port-partitioned rewrite of `population.final.json` that Phase C defines for every
consumer. This entry is that record, so the audit's classification is not silently contradicted and
a future Phase C implementer knows this index exists and predates the general mechanism.

**What changed.** Opening the replacement dialog previously read the complete
`population.final.json` (400,000+ rows in real workspaces) to compute a candidate list that ends up
capped at 100 rows — the single largest avoidable read on the employee-facing path. `getReplacementCandidates()`'s
existing `{ recommended, all }` split (same tier+stage+port = immediate use; broader/cascaded
matches = routed through the existing referral-approval pipeline) is unchanged business logic.

A new replacement-candidate index is built once at population-save time from the `processedRows`
already in memory (zero extra population read), split into up to 10 files per month — one per
(CertScan tier × stage key, including the "unknown" stage) — under
`1-population/{month}/2-processed/replacement-index/`, each holding full `PreparedPopulationRow`

mismatch, or a missing index (legacy months processed before this feature existed), triggers a

full-scan path for the same inputs, not just similar results. The cascade fallback (when the primary
tier+stage bucket is empty after dedup) compares POST-DEDUP supply across sibling buckets, never the
manifest's raw row count — a stage can be almost entirely pre-sampled (e.g. "first" under the
default sampling rules) while still showing a large raw count, which would otherwise pick the wrong
cascade winner. Manifest publication is monotonic (mirrors `updateMonthStatus`'s CAS pattern): a
rebuild is skipped entirely, before any bucket file is touched, if the current manifest's revision is
already at or ahead of it — closing a real bug caught during TDD where a stale rebuild was still
clobbering bucket file *contents* even though the manifest correctly stayed pinned to the newer
revision.

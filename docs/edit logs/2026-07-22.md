## v59.1 — 2026-07-22 — Add: observability breadcrumb for the replacement-candidate index's fallback path

Condition attached to an independent keep/drop review of v59.0 (see that entry below): the index's
greatest strength — every failure mode (missing index, stale revision, stale stage-mappings hash, a
manifest-listed bucket that fails to read) silently degrades to the unchanged full-population scan —
is also a blind spot. If the indexed path never actually worked in a real browser session, nothing
would indicate that; all tests would stay green while every dialog open quietly took the slow path
forever.

`getReplacementCandidatesIndexed` now logs one breadcrumb via the existing error-log ring buffer
(`src/data/storage/errorLogger.ts`, already surfaced to admins through the Settings error view) with
a specific reason (`no-population-revision`, `missing-index`, `stale-index`, or
`bucket-read-failure`) whenever it falls back. Deliberately NOT logged on the success path: logging
every routine indexed read would flood the 50-entry ring buffer with noise on every dialog open and
drown out genuine errors. A month with a working index produces zero entries; a month whose index
never actually works produces one entry per dialog open, which is unmistakable in the error view.

**Files:** `src/data/distribution/{replacementCandidateLookup.ts,replacementCandidateLookup.test.ts}`,
`package.json`, `package-lock.json`

**Before:** A stale/missing/broken replacement-candidate index degraded silently to the full scan
with no trace anywhere that the fast path wasn't being used.

**After:** Every fallback leaves a reasoned breadcrumb in the same error log admins already check,
while a correctly-functioning index stays silent.

## v59.0 — 2026-07-22 — Add: replacement-candidate index so opening the replacement dialog no longer reads the full population

**Governance note — deliberate, recorded exception.** The 2026-07-22 data-management audit
classifies replacement-candidate storage changes as "proposal-covered — Phases A/C/D" of
`docs/architecture/LARGE_POPULATION_PERFORMANCE_PROPOSAL_2026-07-22.md`, and the v57.8 entry below
states the replacement candidate design was "explicitly held for the proposal's approved phase
sequence rather than implemented as ad hoc storage changes." The product owner consciously excepted
this one piece from that hold: it is a narrow, self-contained index for exactly one read path, not
the general port-partitioned rewrite of `population.final.json` that Phase C defines for every
consumer. This entry is that record, so the audit's classification is not silently contradicted and
a future Phase C implementer knows this index exists and predates the general mechanism.

**What changed.** Opening the replacement dialog previously read the complete
`population.final.json` (400,000+ rows in real workspaces) to compute a candidate list that ends up
capped at 100 rows — the single largest avoidable read on the employee-facing path. `getReplacementCandidates()`'s
existing `{ recommended, all }` split (same tier+stage+port = immediate use; broader/cascaded
matches = routed through the existing referral-approval pipeline) is unchanged business logic.

A new replacement-candidate index is built once at population-save time from the `processedRows`
already in memory (zero extra population read), split into up to 10 files per month — one per
(CertScan tier × stage key, including the "unknown" stage) — under
`1-population/{month}/2-processed/replacement-index/`, each holding full `PreparedPopulationRow`
objects (a lightweight projection cannot be used for the immediate-execution path, which splices the
chosen row directly into `sample.master.json`). A small manifest records the source population's
envelope revision and a hash of the fully-resolved stage mappings (stage aliases can be edited
independently of population processing, so the revision alone cannot detect that drift); either
mismatch, or a missing index (legacy months processed before this feature existed), triggers a
fallback to the unchanged full-population read plus a non-blocking background rebuild — never a
population re-save cost on the interactive path.

Bucket contents preserve original population row order (never re-sorted): the existing capSeeded
Fisher-Yates draw is order-sensitive, so this had to produce byte-identical output to the old
full-scan path for the same inputs, not just similar results. The cascade fallback (when the primary
tier+stage bucket is empty after dedup) compares POST-DEDUP supply across sibling buckets, never the
manifest's raw row count — a stage can be almost entirely pre-sampled (e.g. "first" under the
default sampling rules) while still showing a large raw count, which would otherwise pick the wrong
cascade winner. Manifest publication is monotonic (mirrors `updateMonthStatus`'s CAS pattern): a
rebuild is skipped entirely, before any bucket file is touched, if the current manifest's revision is
already at or ahead of it — closing a real bug caught during TDD where a stale rebuild was still
clobbering bucket file *contents* even though the manifest correctly stayed pinned to the newer
revision.

Regression coverage includes an equivalence suite proving the indexed path returns identical
candidate ids in identical order to the pre-existing full-scan path across a normal case, a cascade
case, and an oversized-pool (>100) capping case — the one that actually exercises Fisher-Yates
order-sensitivity — plus the post-dedup-vs-raw-count cascade trap, staleness/fallback triggers, and
index-build failure resilience (a failure here must never sink an otherwise-successful population
save). `replacement.test.ts` passes unmodified, confirming the primitives extracted from it
(`buildExclusionSets`, `isEligibleCandidate`, exported `capSeeded`/`REPLACEMENT_POOL_LIMIT`) preserve
its exact behavior.

**Manual verification note.** The interactive click-through (connect a real workspace → process →
sample → distribute → open the replacement dialog) requires a native OS folder-picker gesture that
browser automation cannot drive — the same limitation noted when reviewing PR #37 earlier today. Verified
instead via the dev server smoke test (clean load, no console errors) and the equivalence test suite,
which directly proves the new path's output matches the old, already-trusted path.

**Pre-commit review.** An independent adversarial review of the full diff (this entry and v58.1
together) found no blockers and two real, non-blocking gaps, both fixed with TDD before commit:

1. The monotonic rebuild guard compared only `sourceRevision`, so an admin editing stage-mapping
   aliases in Settings without reprocessing the month (`sourceRevision` unchanged) permanently blocked
   the self-heal a stale-mappings fallback triggers — every subsequent dialog open would fall back to
   the full scan forever instead of ever rebuilding. Fixed via `isRebuildRedundant()`, which now also
   requires the stage-mappings hash to match before treating a same-revision rebuild as a no-op.
2. `loadReplacementBucket` returning `null` was treated identically whether a (tier, stageKey) bucket
   legitimately had zero rows or the file existed-per-manifest but failed to read — the latter could
   silently under-count candidates instead of falling back to the full scan. Fixed by cross-checking
   each bucket read against the manifest's own bucket list (`loadBucketOrThrowIfExpected`); a
   manifest-listed bucket that fails to read now aborts the indexed path entirely.

Also added direct unit coverage for `memoryDirectory.ts`'s simulated write-permission machinery
(previously only exercised indirectly through consumer tests across v58.1) and strengthened two
`employeeXlsx`/`exportWriter` permission tests to assert the file actually landed on disk, not just
that the call didn't throw.

**Files:** `src/data/population/{replacementIndexTypes.ts,replacementIndexStorage.ts,populationStorage.ts,stageHelpers.ts}`
(+ colocated tests), `src/data/distribution/{replacement.ts,replacementCandidateLookup.ts}` (+ colocated
tests), `src/components/Sidebar/Tabs/EmployeeWorkspace/views/XrayReferrals.tsx`,
`src/data/storage/memoryDirectory.test.ts`, `src/data/answers/employeeXlsx.test.ts`,
`src/data/powerbiExport/exportWriter.test.ts`, `package.json`, `package-lock.json`

**Before:**
```text

any candidates, regardless of how small the eventual capped list turned out to be.
```

**After:**

```text
file(s) matching the dead row's tier and stage — full population reads only occur as a self-healing
fallback for months that predate this index, which then rebuild themselves in the background.
```

## v58.1 — 2026-07-22 — Fix: close the write-permission gap left by the deferred-write workspace fix (v57.5)

v57.5 (PR #36) made the workspace connect read-only by default and defer the write-permission
prompt to the shared `safeWriteJson`/`safeWriteJsonText` boundary. That boundary only covers JSON
writes. Six other entry points create folders or files directly — `saveMonthRun` (the population
save flow), `createBackup`, `restoreBackupSnapshot`, `createDailyAdminBackupIfDue`,
`writeEmployeeXlsx`, and `writeCsvExport` — and none of them requested write access first. On a
freshly-restored remembered workspace, the first population save, backup, restore, or export threw
a raw, English `NotAllowedError` instead of prompting for write access; `createDailyAdminBackupIfDue`
rejected outright since it had no surrounding try/catch.

Routed all six through the existing `withWorkspaceWriteAccess` gate, positioned after any read-only
pre-checks (e.g. the sample-exists guard) so a permission prompt only appears once a write is
actually about to happen. `createDailyAdminBackupIfDue`'s due-check now resolves a clean
`{ ok: false }` instead of rejecting when the workspace can't be read yet, since checking whether a
backup is due should not itself force a write prompt.

Extended `createMemoryDirectory()` (test-only) with a `SimulatedPermissionState` so
`{ create: true }` calls now throw a simulated `NotAllowedError` unless permission was explicitly
granted or requested — closing the test blind spot where every fixture previously hardcoded
`queryPermission`/`requestPermission` to "granted", so this whole class of regression was
untestable. Added a `setSimulatedWritePermission()` test helper for flipping an already-connected
handle's permission mid-test (e.g. simulating a new session reconnecting the same workspace
read-only). 10 new regression tests cover both the granted-after-prompt and declined paths for all
six entry points.

Found during a 2026-07-22 review of PR #36 ("Remember workspace and defer write permission").

**Files:** `src/data/population/populationStorage.ts`, `src/data/backup/backupStorage.ts`,
`src/data/answers/employeeXlsx.ts`, `src/data/powerbiExport/exportWriter.ts`,
`src/data/storage/memoryDirectory.ts`, and their colocated tests, plus `package.json` and
`package-lock.json`


## v57.9 — 2026-07-22 — Fix (referrals): make supervisor reassignment approval source-true and interruption-safe

Supervisor denial succeeded while approval could appear to do nothing because the approval-only
ownership check trusted `distribution.current.json`. That file is a rebuildable cache, and a
stale-but-valid snapshot could disagree with the immutable distribution event log and incorrectly
report that employee A no longer owned the samples. Approval now derives ownership from the fold
of the already-loaded event log, which is the authoritative state, and reloads and folds the
persisted events again to verify employee B owns every pending sample before saving the supervisor
decision. Neither `distribution.current.json` nor the compatibility log projection is mutated
directly; transfers still go exclusively through `appendDistributionEvents` and the legal fold
transitions.

The replay guard also treated any event carrying the request ID as proof that an entire
multi-sample reassignment had completed. Immutable event files are independently durable, so an
interruption after one file could leave a partial batch; retry would then skip the missing samples
and incorrectly record approval. Replay is now idempotent per `xrayImageId`: it recognizes only
matching reassignment events to the requested employee, appends only missing events, and verifies
the complete folded transfer before recording approval. This is safe under arbitrary interruption
points, including a failure between any two immutable event-file writes or after the transfer but
before the decision write.

The supervisor confirmation modal now sets a synchronous single-submit guard before starting the
async write, disables its controls, and shows a configurable Arabic busy label. A slow workspace
disk therefore cannot double-fire `appendDistributionEvents` during the pre-render double-click
window. Regression coverage uses `createMemoryDirectory()` for stale-cache ownership and a
one-of-two-files-persisted retry, plus a slow unresolved UI callback that verifies two clicks invoke
the approval only once.

**Files:** `src/data/referral/{approveReferral.ts,approveReferral.test.ts}`,
`src/components/Sidebar/Tabs/EmployeeWorkspace/views/ReferralApproval/{ReviewModal.tsx,ReviewModal.test.tsx,index.tsx}`,

## v57.8 — 2026-07-22 — Docs: audit app data management boundaries

Audited the monthly data lifecycle against the intended employee experience and documented the
implementation boundary with the proposed large-population architecture. The audit confirms that
new-month processing state resets without deleting historical month folders, while identifying
four gaps: employee defaults and landing order expose Population and extra Employee Workspace
pages; X-ray Referrals reads the complete sample master and distribution before filtering despite
an existing per-employee mirror; replacement reads the complete processed population, currently
caps the returned list at 100 after an earlier 1,000-row cap, and has no precomputed reserve; and
the global month state has no all-month selection kind.

Each finding is classified as independently decidable now or covered by the pending performance
proposal. The employee read path and replacement candidate design are explicitly held for the
proposal's approved phase sequence rather than implemented as ad hoc storage changes. All-month
semantics may be defined separately, but heavy cross-month aggregation remains within the scalable
query work and all mutations must retain one explicit month target.

The audit also records the open policy decision for employee permissions already persisted in
browser `localStorage`: leave existing installations untouched, run a conservative one-time
migration, or introduce versioned defaults with inherited/custom provenance. It recommends
versioned defaults, optionally combined with an idempotent migration limited to values proven to
match the previous shipped defaults, so administrator overrides are never silently replaced.

## v57.4 — 2026-07-22 — Fix (backup): make restorable backups JSON-first and bound optional XLSX memory

Owner-reported: workspace backups could take an extremely long time and sometimes fail with
`Invalid array length`. The backup path copied the complete authoritative JSON snapshot and then,
before declaring success, loaded every month again to rebuild redundant XLSX copies in 250,000-row
SheetJS workbooks. Real workspaces exceed 400,000 rows. SheetJS materializes worksheet cells and
the XLSX ZIP in browser memory, so wide datasets could require millions of live cell objects plus
serialization buffers and exhaust the JavaScript array/heap limits.

**Behavior change:** routine manual, automatic, and pre-restore backups now create the complete,
restorable JSON snapshot only. XLSX files are convenience exports, not restore inputs, and are no
longer generated silently as part of every backup. The Archive screen exposes a clearly labeled,
default-off Arabic checkbox for users who explicitly need XLSX copies of smaller datasets.

The compatibility XLSX path now limits a dataset to 100,000 rows, limits each workbook part to
25,000 rows and approximately 250,000 cells, and collects headers incrementally instead of using a
large `flatMap` allocation. Oversized or failed XLSX exports produce a clear Arabic warning while
the completed JSON backup is still recorded successfully. Existing backup manifests retain their
XLSX fields (empty in new JSON-only backups), and history continues to read legacy manifests with
populated XLSX metadata.

Regression coverage verifies the JSON-only default, small explicit XLSX exports, immediate
400,000-row rejection before worksheet allocation, missing-month behavior, and legacy manifest
history compatibility.

**Files:** `src/data/backup/backupStorage.ts`,
`src/data/backup/backupStorage.test.ts`,
`src/components/Sidebar/Tabs/Archive/index.tsx`,
`src/components/Sidebar/Tabs/Archive/Archive.css`,
`src/data/labels/labelsStore.ts`,
`docs/architecture/data-system-report.md`, `package.json`, and `package-lock.json`

**Before:**
```text
Every backup synchronously regenerated all month-scale XLSX datasets in 250,000-row in-memory
workbooks; a spreadsheet allocation failure also caused the restorable backup operation to fail.
```

**After:**

# Desktop App-Mode Shortcut — Design Spec

- **Date:** 2026-07-16
- **Status:** Approved by user (2026-07-16)
- **Scope:** Give the existing single-file build an installed-app look and feel (own window, no browser tabs/address bar, taskbar/desktop icon) without a service worker, a manifest, or a second build output.

## Background

The user initially asked for `npm run build` to produce two outputs: the existing portable single-file static build, and a second PWA (installable, offline-capable) build in a separate folder.

Investigation surfaced a hard technical constraint: a service worker — what makes a PWA installable and offline-capable — refuses to run on `file://` URLs; it requires an HTTPS or `http://localhost` origin. The user's actual requirement ("no server at all," confirmed in discussion) rules out that path entirely — no workaround exists.

Further, a true PWA's other headline benefit (offline caching of network responses) is moot for this app: it has zero backend and zero network calls — all data I/O goes through the local File System Access API. So a full PWA build would add real complexity (bundled local server, service worker, web manifest, second build output folder) for a benefit set that reduces, in practice, to one thing: an installed-app look.

The user chose to get that one thing directly, at near-zero cost: a Windows shortcut that launches the browser in **app mode** (`--app=file:///...`) against the *existing* single-file build. This opens the app in its own window with no tabs, no address bar, and a custom icon — visually indistinguishable from an installed PWA to an end user — with zero changes to the build pipeline itself.

**Explicitly not pursued and why:** a full PWA build (service worker + manifest + bundled local server + second `dist-pwa/` output) — rejected because it would only add network-offline-caching (nothing to cache) and OS-level "installed apps" registration, neither of which the user needs, in exchange for meaningfully more moving parts (a server process, a second build target, new failure modes).

## Design

### Files added (all under `public/`)

Vite's existing convention copies everything in `public/` into `dist/` verbatim on every build — no `vite.config.ts` change needed.

| File | Purpose |
|------|---------|
| `public/app-icon.ico` | Multi-resolution icon (16/32/48/256px), generated once from the SVG already inlined as the app's favicon in `index.html` (navy X-ray-film mark with a teal checkmark badge). A static, checked-in asset — never regenerated by the build. |
| `public/create-desktop-shortcut.ps1` | The shortcut-creation logic (see below). |
| `public/Create Desktop Shortcut.bat` | One-click wrapper: `powershell -ExecutionPolicy Bypass -NoProfile -File "%~dp0create-desktop-shortcut.ps1"`. Exists so a non-technical user isn't blocked by PowerShell's default script-execution policy, which normally refuses to run a bare `.ps1` on double-click.

After `npm run build`, `dist/` contains these three files alongside `index.html`, automatically, on every build.

### Deployment flow

1. IT/admin copies the whole `dist/` folder to wherever it's going to live (a shared drive, a user's machine, a USB stick — unchanged from today).
2. The end user (or IT, once, per machine) double-clicks `Create Desktop Shortcut.bat` inside that folder.
3. A shortcut named "نظام معالجة بيانات الأشعة" appears on the Desktop. From then on the user launches the app from that shortcut instead of double-clicking `index.html` directly.

Both launch paths keep working — the plain `index.html` double-click is untouched; the shortcut is additive.

### `create-desktop-shortcut.ps1` behavior

1. **Self-locate:** resolves `$scriptDir` from its own invocation path (`$MyInvocation.MyCommand.Path`), so the shortcut always points at wherever this copy of `dist/` actually is — never a path baked in at build time.
2. **Verify `index.html` exists** next to it. If missing, print a clear Arabic error to the console and exit non-zero — no shortcut created pointing at nothing.
3. **Detect a Chromium browser**, in order:
   - Chrome: `%ProgramFiles%\Google\Chrome\Application\chrome.exe`, `%ProgramFiles(x86)%\Google\Chrome\Application\chrome.exe`, `%LocalAppData%\Google\Chrome\Application\chrome.exe`
   - Edge (fallback — ships with every Windows install; CLAUDE.md already documents both Chrome and Edge as supported): `%ProgramFiles%\Microsoft\Edge\Application\msedge.exe`, `%ProgramFiles(x86)%\Microsoft\Edge\Application\msedge.exe`
   - If neither is found: print an Arabic error naming the requirement (a Chromium browser must be installed) and exit non-zero. No shortcut created.
4. **Create the shortcut** via the `WScript.Shell` COM object (`CreateShortcut`), targeting the Desktop folder (`[Environment]::GetFolderPath("Desktop")`):
   - `TargetPath` = the found browser executable
   - `Arguments` = `--app="file:///<resolved absolute path>/index.html" --new-window`
   - `IconLocation` = the resolved absolute path to `app-icon.ico`
   - `WorkingDirectory` = the resolved `dist` folder
   - `Description` = "نظام معالجة بيانات الأشعة"
   - Shortcut file name: `نظام معالجة بيانات الأشعة.lnk`
5. **Idempotent:** re-running overwrites the same shortcut file — safe to re-run after moving the `dist/` folder to a new location (e.g., an app update), which naturally "repairs" a stale shortcut.
6. Prints a success message on completion.

### Icon asset generation

`app-icon.ico` is generated once (not part of any build step) from the existing inline SVG favicon in `index.html`, at 16/32/48/256px, and committed to the repo under `public/`. If the app's branding SVG ever changes, this file is regenerated manually and re-committed — it is not derived automatically at build time.

### Explicitly out of scope

- No service worker, no web app manifest, no offline network-response caching (nothing to cache — the app makes no network calls).
- No bundled local HTTP server, no second build output folder (e.g. no `dist-pwa/`).
- No new npm dependencies; `npm run build` / `npm run dev` / `npm run preview` are all unchanged.
- No OS-level "installed apps" registration (Windows Settings → Apps) — this is a browser-app-mode shortcut, not a true installed PWA. The user explicitly chose this tradeoff after the cost/benefit discussion.

## Edge cases

- **Neither Chrome nor Edge installed:** script errors out clearly rather than creating a shortcut that won't launch.
- **`dist/` folder moved after the shortcut was created:** the old shortcut breaks (points at a stale path) until the user re-runs `Create Desktop Shortcut.bat` from the new location, which overwrites it with the correct path (see "Idempotent" above).
- **Re-running on the same machine/location:** safe no-op that just recreates an identical shortcut.
- **Non-Windows deployment:** out of scope — the app already requires a Chromium browser and, per CLAUDE.md, the File System Access API restricts real usage to Windows/Chromium environments in practice; this feature assumes that same environment.
- **Paths containing spaces or parentheses:** deployment folder names are not guaranteed to be simple (this very repo's own path contains both, e.g. `...\New folder (2)\x-ray-quality-app-v1`). The `Arguments` value must be built and quoted so the full `file:///` URI survives as one token to the browser regardless of spaces/parens in the path — verified by testing shortcut creation from a folder with spaces and parentheses in its name, not just a clean path.

## Testing

This is a deployment-tooling feature (a PowerShell script + a static icon), not application code — it has no unit-testable logic in the Vitest sense. Verification is manual: run `npm run build`, confirm the three files land in `dist/`, then (on a Windows machine) double-click `Create Desktop Shortcut.bat` and confirm the resulting shortcut launches the app in app-mode with the correct icon.
